Tic-Tac-Toe with RewriteGames.jl

From algebraic rewriting rules to a trained ONNX policy

Author

RewriteGames.jl contributors

Published

April 21, 2026

Overview

RewriteGames.jl models game states as attributed C-sets (ACSets) and moves as algebraic rewrite rules. Both concepts come from applied category theory: ACSets are typed relational data structures whose schemas describe objects (tables) and morphisms (foreign keys), and rewrite rules describe how those structures transform under play.

This choice of representation is deliberately general. Any game whose state can be described as a relational structure — boards, graphs, maps, unit networks — fits naturally into the framework. Because every game shares the same substrate, the engine’s bookkeeping (match enumeration, budget tracking, serialisation) applies uniformly across games without modification.

Two longer-term goals shape the library’s design, though neither is fully realised yet:

  • Modular game composition. ACSets and rewrite rules support algebraic composition: schemas can be combined, rules can be glued together, and schedules can be nested. The infrastructure for this — GameMigration, compositional schedules — is in place; a library of reusable primitives is forthcoming.
  • Cross-game transfer. Because games share a common representation, a policy trained on one game could, in principle, transfer to another via schema migration. This is ongoing research; the current library provides the representational foundation but not yet the transfer machinery.

In this tutorial we use tic-tac-toe to demonstrate every layer of the framework: simple enough to understand at a glance, rich enough to exercise the full workflow.

  1. Model the board as an ACSet and the legal moves as algebraic rewrite rules.
  2. Run self-play episodes and serialise the experience stream to Arrow.
  3. Train a policy network in PyTorch with REINFORCE.
  4. Export the policy to ONNX and load it back into Julia as a drop-in agent.

Part 1 — Modelling the Game in Julia

Schema design

An ACSet (attributed C-set) is a relational data structure whose schema describes the tables and foreign keys. For tic-tac-toe we use three tables:

Table Meaning
Sq Board squares (always exactly 9)
X X pieces placed so far
O O pieces placed so far

Two foreign keys record which square each piece occupies:

@present SchTTT(FreeSchema) begin
    Sq::Ob
    X::Ob
    O::Ob
    xsq::Hom(X, Sq)
    osq::Hom(O, Sq)
end

@acset_type TicTacToe(SchTTT, index=[:xsq, :osq])
graph LR
    X["X  (pieces)"] -->|xsq| Sq["Sq  (squares)"]
    O["O  (pieces)"] -->|osq| Sq

We index on both foreign keys so that incident(board, sq_id, :xsq) — “all X pieces on square sq_id” — is an O(1) lookup.

Squares are numbered 1–9 in row-major order:

 1 │ 2 │ 3
───┼───┼───
 4 │ 5 │ 6
───┼───┼───
 7 │ 8 │ 9

The initial world has 9 squares and no pieces:

function initial_board()
    b = TicTacToe()
    add_parts!(b, :Sq, 9)
    return b
end

Game schedule

Every game in RewriteGames is driven by a schedule — a tree of GameStep nodes that describes one round of play. The driver executes the tree once per round and loops until the terminal predicate fires or T_max steps are reached.

The schedule determines who can act and when. Each PlayerStep asks the named player’s agent to pick from their available rewrite rules and apply one. Because the schedule is explicit, turn order, conditional branching, and per-unit actions are all expressible without modifying the engine.

For tic-tac-toe, one round is simply “X moves, then O moves”:

Seq(PlayerStep(:X), PlayerStep(:O))

Seq runs its children in order and short-circuits as soon as the game ends — if X wins mid-round, O never moves.

The full set of node types is:

Type Purpose
PlayerStep(p) Player p selects and applies one action
AutoStep() Fire game.auto rules without player input
Seq(steps...) Run children in order; short-circuit on terminal
Cond(pred, branches...) Branch on pred(W) -> Int (1-based index)
WhileStep(cond, body) Repeat body while cond(W) is true
ForEachStep(ob, body) Iterate over every part of type ob

Each Experience record carries a schedule_path vector that identifies which nodes were active when it was emitted — useful for logging, replay, and attributing steps in multi-agent or hierarchical schedules.

Note

PlayerStep does not automatically fire auto-rules; that is AutoStep’s responsibility. For games with probabilistic environment events, interleave AutoStep() nodes explicitly in the schedule. Tic-tac-toe has no auto-rules, so the two-node Seq above is the complete schedule.

Rewrite rules

In tic-tac-toe, each player’s only legal move is to place their piece on an empty square. We model this as an algebraic rewrite rule on the board ACSet: find an empty square (the match), then add a piece pointing at it (the rewrite).

A rewrite rule in AlgebraicRewriting is a span L ← I → R of ACSet morphisms:

  • L (left): the pattern that must be found in the current world.
  • I (interface): the sub-pattern of L that is preserved by the rewrite.
  • R (right): what the matched region becomes after the rewrite.

The engine finds all injective morphisms m : L → G from the pattern L into the current world G that satisfy every attached condition. For each valid match it applies the double-pushout (DPO) construction to produce the updated world G'.

Pattern and interface

Both the “place X” and “place O” rules search for the same thing: one bare square. We build the shared pattern and interface once:

# Shared pattern and interface: one bare square
I_sq = @acset TicTacToe begin Sq=1 end
L_sq = @acset TicTacToe begin Sq=1 end

l_sq = ACSetTransformation(I_sq, L_sq; Sq=[1])   # identity on Sq
graph LR
    subgraph L["L — find a square"]
        LSq["Sq"]
    end
    subgraph I["I — keep the square"]
        ISq["Sq"]
    end
    subgraph R["R — square + new piece"]
        RSq["Sq"]
        RX["X  (or O)"]
        RX -->|"xsq"| RSq
    end
    ISq -.->|l| LSq
    ISq -.->|r| RSq

This span alone would match every square on the board, including occupied ones. We need to restrict matches to squares that are genuinely empty.

Negative Application Conditions

A Negative Application Condition (NAC) is a morphism f : L → N that forbids matches. Formally, a match m : L → G is only legal if there is no morphism n : N → G satisfying n ∘ f = m. In plain English: the engine checks whether the larger pattern N can be embedded in the world consistently with the proposed match. If it can, the match is rejected.

For tic-tac-toe, N is “a square with a piece already on it”. We build one NAC forbidding X-occupied squares and one forbidding O-occupied squares:

# Forbidden patterns: a square that already holds a piece
N_x_occ = @acset TicTacToe begin Sq=1; X=1; xsq=[1] end  # square with an X
N_o_occ = @acset TicTacToe begin Sq=1; O=1; osq=[1] end  # square with an O

# Morphisms send the bare L_sq square to the square inside each forbidden pattern
nac_f_x = ACSetTransformation(L_sq, N_x_occ; Sq=[1])
nac_f_o = ACSetTransformation(L_sq, N_o_occ; Sq=[1])

nac_no_x = NAC(nac_f_x)   # reject if matched square already holds an X
nac_no_o = NAC(nac_f_o)   # reject if matched square already holds an O
graph LR
    subgraph L["L — candidate square"]
        LSq["Sq"]
    end
    subgraph N["N — forbidden: square with X"]
        NSq["Sq"]
        NX["X"]
        NX -->|xsq| NSq
    end
    LSq -.->|f| NSq

Attaching both nac_no_x and nac_no_o to a rule means get_matches automatically returns only squares that are empty in both respects. The game engine never sees an illegal move candidate; no post-hoc filtering is needed.

Completing the rules

With the shared interface and NACs in hand, building the two rules is straightforward:

# Place X on the matched square
R_x  = @acset TicTacToe begin Sq=1; X=1; xsq=[1] end
r_x  = ACSetTransformation(I_sq, R_x; Sq=[1])
rule_place_x = Rule(l_sq, r_x; ac=[nac_no_x, nac_no_o])

# Place O on the matched square
R_o  = @acset TicTacToe begin Sq=1; O=1; osq=[1] end
r_o  = ACSetTransformation(I_sq, R_o; Sq=[1])
rule_place_o = Rule(l_sq, r_o; ac=[nac_no_x, nac_no_o])

Both rules forbid playing on a square that holds either piece — only a square that is empty in both respects produces a valid match.

Terminal predicate

We check the eight winning lines (3 rows, 3 columns, 2 diagonals):

const WINS = [
    [1,2,3], [4,5,6], [7,8,9],    # rows
    [1,4,7], [2,5,8], [3,6,9],    # columns
    [1,5,9], [3,5,7],             # diagonals
]

function check_terminal(board)
    xs = Set(subpart(board, parts(board, :X), :xsq))
    os = Set(subpart(board, parts(board, :O), :osq))

    for line in WINS
        all(s ∈ xs for s in line) && return (true, :X)
        all(s ∈ os for s in line) && return (true, :O)
    end

    nparts(board, :X) + nparts(board, :O) == 9 && return (true, nothing)
    return (false, nothing)
end

Assembling the game with @game

RewriteGames exports a @game DSL macro that builds a Game from a concise block. Keyword clauses map to the Game(...) constructor arguments:

# No post_filter needed — the NACs embedded in the rules already ensure that
# get_matches only returns empty-square matches.
#
# RuleEntry also accepts an optional `budget=N` keyword that caps how many
# times the rule may fire per episode.  Tic-tac-toe needs no cap (each player
# can place at most 5 pieces naturally), but it is useful for more complex
# games where rule-use limits encode game semantics.
entry_x = RuleEntry(rule_place_x; name=:place)
entry_o = RuleEntry(rule_place_o; name=:place)

game = @game nothing begin
    players:  X, O
    X:        [entry_x]
    O:        [entry_o]
    terminal: check_terminal
    initial:  initial_board
    schedule: Seq(PlayerStep(:X), PlayerStep(:O))
end
Note

initial: initial_board passes the function object itself (no parentheses). The engine calls game.initial() at the start of each episode.

schedule: Seq(PlayerStep(:X), PlayerStep(:O)) makes the round structure explicit. Without a schedule: clause the engine auto-generates Seq(PlayerStep(p), AutoStep()) for each player in order, which is equivalent for this game — but supplying the schedule explicitly is clearer and is the recommended style.

Displaying the board

A quick ASCII helper returns the board as a string:

function show_board(board) :: String
    marks = fill(' ', 3, 3)
    for xp in parts(board, :X)
        sq = subpart(board, xp, :xsq)
        marks[(sq-1)÷3+1, (sq-1)%3+1] = 'X'
    end
    for op in parts(board, :O)
        sq = subpart(board, op, :osq)
        marks[(sq-1)÷3+1, (sq-1)%3+1] = 'O'
    end
    sep = "───┼───┼───"
    rows = String[]
    for (i, row) in enumerate(eachrow(marks))
        push!(rows, " " * join(string.(row), " │ "))
        i < 3 && push!(rows, sep)
    end
    return join(rows, "\n")
end

Part 2 — Collecting Training Data

Before training in Python we need to serialise the board state and chosen action for every step.

Built-in vs custom serialisation

RewriteGames exports write_experiences / read_experiences (backed by Arrow) that save the generic ACSet encoding from every Experience — node_features, edge_index, rule counters, and the chosen rule name. This is convenient for graph-neural-network policies that consume the raw ACSet graph.

Here we write a game-specific writer instead, because our policy network expects a fixed 27-element one-hot feature vector (9 squares × 3 states) that is cheaper and more interpretable than the generic encoding. Whenever your policy can work with the generic ACSet encoding, prefer write_experiences.

Feature extraction

We represent each board as a 9 × 3 one-hot matrix (rows = squares, columns = [empty, X, O]) and flatten it to a 27-element Float32 vector:

function board_features(board) :: Vector{Float32}
    feats = zeros(Float32, 9, 3)
    xs = Set(subpart(board, parts(board, :X), :xsq))
    os = Set(subpart(board, parts(board, :O), :osq))
    for sq in 1:9
        if sq ∈ xs
            feats[sq, 2] = 1f0
        elseif sq ∈ os
            feats[sq, 3] = 1f0
        else
            feats[sq, 1] = 1f0
        end
    end
    return vec(feats)
end

Custom experience writer

We collect (features, chosen_square, player, outcome) tuples across many episodes and write them to an Arrow file:

struct StepRecord
    features       :: Vector{Float32}  # 27-element board encoding
    chosen_square  :: Int32            # 1–9; 0 means the player passed
    player         :: String           # "X" or "O"
    outcome        :: Float32          # +1 win, -1 loss, 0 draw/timeout
end

function collect_episodes(game, agents; n_episodes=5_000, T_max=20)
    records = StepRecord[]

    for _ in 1:n_episodes
        episode = run_game(game, agents; T_max=T_max)
        final   = episode[end]
        winner  = final.winner       # :X, :O, or nothing

        for exp in episode
            exp.action === nothing && continue   # skip passes

            sq    = Int32(components(exp.action.match)[:Sq][1])
            board = exp.state.raw.world
            feats = board_features(board)
            out   = if winner === exp.player; 1f0
                    elseif winner === nothing;  0f0
                    else                       -1f0 end

            push!(records, StepRecord(feats, sq, string(exp.player), out))
        end
    end
    return records
end

Run the collector with X playing randomly and O playing randomly, then save:

Random.seed!(0)

Data loading

#| label: py-load-data
def load_training_data(path: str):
    """Load the Arrow file written by Julia."""
    with ipc.open_file(path) as reader:
        tbl = reader.read_all()

    features = np.stack(tbl["features"].to_pylist()).astype(np.float32)  # (N, 27)
    squares  = np.array(tbl["chosen_square"].to_pylist(), dtype=np.int64) - 1  # 0-indexed
    players  = np.array(tbl["player"].to_pylist())
    outcomes = np.array(tbl["outcome"].to_pylist(), dtype=np.float32)

    return features, squares, players, outcomes

Policy network

The network takes the 27-dimensional board encoding and outputs a logit for each of the 9 squares. The agent will mask the illegal squares (already occupied) at inference time — the network never needs to worry about them during training either, since the data only contains legal moves.

#| label: py-network
class TicTacToeNet(nn.Module):
    """MLP policy: board features (27) → square logits (9)."""

    def __init__(self, hidden: int = 128):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(27, hidden),
            nn.ReLU(),
            nn.Linear(hidden, hidden),
            nn.ReLU(),
            nn.Linear(hidden, 9),
        )

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # x: (batch, 27)  →  logits: (batch, 9)
        return self.net(x)

REINFORCE training loop

We use the Monte Carlo policy-gradient (REINFORCE) algorithm. Every recorded step carries its final outcome as the return signal, so the update is simply:

\[\nabla_\theta J \approx \sum_t G_t \, \nabla_\theta \log \pi_\theta(a_t \mid s_t)\]

where \(G_t \in \{-1, 0, +1\}\) is the game result for the player who moved at step \(t\).

#| label: py-train
def train(
    model: nn.Module,
    features: np.ndarray,
    squares: np.ndarray,
    outcomes: np.ndarray,
    *,
    epochs: int = 80,
    batch_size: int = 512,
    lr: float = 3e-4,
    baseline: bool = True,          # subtract a running mean baseline
):
    optimiser = torch.optim.Adam(model.parameters(), lr=lr)
    n = len(features)
    running_baseline = 0.0

    for epoch in range(epochs):
        idx = np.random.permutation(n)
        epoch_loss = 0.0
        batches    = 0

        for start in range(0, n, batch_size):
            batch_idx = idx[start : start + batch_size]
            x   = torch.from_numpy(features[batch_idx])
            a   = torch.from_numpy(squares [batch_idx])
            G   = torch.from_numpy(outcomes[batch_idx])

            if baseline:
                with torch.no_grad():
                    running_baseline = 0.9 * running_baseline + 0.1 * G.mean().item()
                G = G - running_baseline

            logits = model(x)                    # (batch, 9)
            log_prob = F.log_softmax(logits, dim=-1)
            chosen_lp = log_prob[torch.arange(len(a)), a]  # log π(aₜ | sₜ)

            loss = -(chosen_lp * G).mean()       # negative for gradient ascent

            optimiser.zero_grad()
            loss.backward()
            optimiser.step()

            epoch_loss += loss.item()
            batches    += 1

        if (epoch + 1) % 10 == 0:
            print(f"Epoch {epoch+1:3d}  loss={epoch_loss/batches:.4f}")

    return model


torch.manual_seed(0)
model = TicTacToeNet(hidden=128)
model = train(model, features, squares, outcomes, epochs=80)

Evaluating the trained policy

We run a quick greedy evaluation against a random opponent, applying the illegality mask:

#| label: py-eval
def greedy_move(model, features_np, legal_mask):
    """Return the argmax-policy square index (0-indexed), masking illegal squares."""
    x = torch.from_numpy(features_np[None, :])
    with torch.no_grad():
        logits = model(x).squeeze(0)            # (9,)
    logits[~legal_mask] = -1e9                  # mask occupied squares
    return int(logits.argmax())


def evaluate(model, n_games: int = 500, seed: int = 1):
    """
    Play `n_games` of X (policy) vs O (random).
    Returns (x_wins, draws, o_wins).
    """
    rng = np.random.default_rng(seed)
    x_wins = draws = o_wins = 0

    for _ in range(n_games):
        board = np.zeros(9, dtype=np.int8)   # 0=empty, 1=X, 2=O

        for turn in range(9):
            player = 1 if turn % 2 == 0 else 2   # X=1, O=2 moves first
            empty  = np.where(board == 0)[0]

            if player == 1:
                # one-hot features: col0=empty, col1=X, col2=O
                feats = np.zeros(27, dtype=np.float32)
                for sq in range(9):
                    feats[sq * 3 + board[sq]] = 1.0
                mask = torch.zeros(9, dtype=torch.bool)
                mask[empty] = True
                sq = greedy_move(model, feats, mask)
            else:
                sq = int(rng.choice(empty))

            board[sq] = player

            # Check win
            wins = [(0,1,2),(3,4,5),(6,7,8),
                    (0,3,6),(1,4,7),(2,5,8),
                    (0,4,8),(2,4,6)]
            for a,b,c in wins:
                if board[a] == board[b] == board[c] == player:
                    if player == 1: x_wins += 1
                    else:           o_wins += 1
                    board = None   # signal game over
                    break
            if board is None:
                break
        else:
            draws += 1

    return x_wins, draws, o_wins


x_wins, draws, o_wins = evaluate(model, n_games=500)
print(f"X (policy) wins: {x_wins/5:.1f}%  "
      f"Draws: {draws/5:.1f}%  "
      f"O (random) wins: {o_wins/5:.1f}%")

def export_onnx(model: nn.Module, path: str = "ttt_policy.onnx"):

Part 4 — Loading the ONNX Policy in Julia

Now we load ttt_policy.onnx back into Julia using the ONNXAgentExt package extension. The extension is activated automatically when you load ONNXRunTime alongside RewriteGames.

EncodedState — what agents receive

Every agent’s select_action method receives an EncodedState, which bundles the raw game state with pre-computed tensor encodings:

Field Type Contents
raw GameState Raw world ACSet plus counters and turn number
node_features Matrix{Float32} (n × F) One-hot table type + attribute values per (table, row) node
edge_index Matrix{Int32} (2 × E) COO-format foreign-key edges
edge_type Vector{Int8} Schema morphism index per edge
rule_counters Vector{Int32} Remaining budget per rule
turn_frac Float32 turn / T_max

state.raw.world is the live ACSet. state.node_features and state.edge_index are already in a format suitable for graph neural networks. Simple agents that only need the raw board can access state.raw.world directly and pay no encoding cost.

Building the input function

ONNXAgent calls input_fn(state::EncodedState) before every inference step. For tic-tac-toe we ignore the generic ACSet encoding and instead build our fixed-size feature vector directly from the raw board:

function ttt_input_fn(state::EncodedState)
    board = state.raw.world
    feats = board_features(board)              # 27-element Float32 vector
    return Dict("board" => reshape(feats, 1, :))  # shape (1, 27) as expected by ONNX
end

A square-aware policy wrapper

RewriteGames exports ONNXAgent as a built-in agent type for models whose output logits align directly with positions in legal_actions. Tic-tac-toe does not fit that pattern because our model outputs one logit per square (1–9), while legal_actions is a variable-length vector of only the empty squares. We therefore write a thin FunctionAgent wrapper that translates square logits into the correct Action:

using ONNXRunTime

function make_ttt_agent(model_path::String)
    session = ONNXRunTime.load_inference(model_path)

    policy = function (state::EncodedState, legal_actions::Vector{Action})
        # Run ONNX inference
        inputs  = ttt_input_fn(state)
        outputs = session(inputs)
        logits  = vec(first(values(outputs)))   # length-9 Float32 vector

        # Map each legal action to its square, pick the highest-logit legal square
        legal_sq  = [components(a.match)[:Sq][1] for a in legal_actions]
        best_idx  = argmax([logits[sq] for sq in legal_sq])
        return legal_actions[best_idx]
    end

    return FunctionAgent(policy)
end

function benchmark(game, x_agent, o_agent; n=500, T_max=20, seed=1)
    Random.seed!(seed)
    agents = Dict{Symbol, AbstractAgent}(:X => x_agent, :O => o_agent)
    episodes = [run_game(game, agents; T_max=T_max) for _ in 1:n]
    all_exps = reduce(vcat, episodes)

    x_wr = win_rate(all_exps, :X)
    o_wr = win_rate(all_exps, :O)
    draw = 1 - x_wr - o_wr
    println("Over $n games:")
    println("  X (trained) wins: $(round(100x_wr; digits=1))%")
    println("  draws:            $(round(100draw;  digits=1))%")
    println("  O (random) wins:  $(round(100o_wr; digits=1))%")
end

RewriteGames.jl handles the combinatorial bookkeeping (pattern matching, budgets, auto-rules, serialisation) so you can focus on game semantics and learning algorithms.

Going further

  • Self-play curriculum: use a FunctionAgent that randomly mixes the trained policy with random moves, then progressively reduce the random fraction.
  • Value network: add a value head to the network for actor-critic methods; the EncodedState.node_features tensor from RewriteGames is already GNN-compatible (COO format + node features).
  • More complex games: the same schema pattern extends to hex, go, or any game whose state is naturally a graph — the rewrite-rule formulation handles arbitrary topology changes.
  • Rich schedules: use Cond, WhileStep, or ForEachStep in the schedule tree to model games with variable turn order, per-unit actions (e.g. each piece moves independently), or conditional phases. The schedule_path in each Experience record tells you exactly where in the tree that step was emitted.
  • Auto-rules: wrap any rule in AutoRule(rule; prob_attr=:p) and add it to the auto: clause of @game. Auto-rules fire automatically between player steps via AutoStep() nodes in the schedule, making it easy to model stochastic environment dynamics (weather, spawning, decay).
  • Schema migration: GameMigration lets you apply a Catlab schema functor to an entire game (world, rules, and initial factory) in one call. This is useful when you want to extend a prototype schema with new attributes or objects without rewriting all the rules from scratch.
